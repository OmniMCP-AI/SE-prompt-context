## 1) What the current pipeline actually does

Current production-like chain is a **workflow-of-workflows orchestrator**.

Main orchestrator JSON:
- `key-workflow/ecommerce/step4-groupbyspu.json`
- same structure also appears in `multiple-workflow-steps.json`

Orchestrator steps (sequential):
1. `update_origin_weekly_data` → artifact `...be57`
2. `get_main_sku` → artifact `...be5a`
3. `calcu_weekly_data` → artifact `...be5d`
4. `calcu_team_category_data` → artifact `...be60`
5. `split_team_category_data` → artifact `...be63`

This orchestrator itself does not output final tables directly. Each step calls a sub-workflow by `workflow_mcp__run_workflow`.

## 2) Major sub-workflow logic (from local JSON snapshots)

## Stage S1: Source sheet sync + formula recalc
File: `step1-prepare-update.json`

- Reads raw sheets (`联盟`, `订单`, `广告`, `ERP`, `商业分析`, `日期表`, etc.)
- Writes/overwrites corresponding target sheets with `excel__update_data_keep_headers`
- Runs `excel__recalculate_formulas` on final workbook

Meaning:
- This stage is a **copy/sync stage** from source workbook tabs to target workbook tabs
- It preserves template headers/formulas and refreshes data blocks

## Stage S2: Main key extraction + formula fill
File: `step2-prepare-uniq-spu.json`

- Reads `订单`
- SQL dedupe/build KID (`店铺-SPU分类-主货号`)
- Writes to `周数据-all` at `D2`
- Autofills formulas via `excel__copy_range_with_formulas` (`I2:U2`, lookup by column `G`)
- Extracts unique `商品编号` and writes to `链接数据统计` at `B2`
- Autofills link-sheet formulas from template range (`C2:AD2`, lookup by `B`)

Meaning:
- This stage creates primary keys and drives row expansion for formula-based report rows

## Stage S3: Pivot/aggregation into PT sheets + recalc
File: `step3-bronze-sivler1-pivit-table.json` (and older `pivot-table.json` equivalent)

Reads raw business sheets and aggregates with SQL:
- `订单` → `订单中间表-PT` (order count, quantity by KID)
- `ERP` → `ERP-主货号-PT` (sales, fees, costs by KID)
- `联盟` → `联盟-PT` (alli cost by KID)
- `广告` → `广告-PT` (ad cost by KID, MY/TH multipliers)
- `大盘` → `大盘-PT` (visit/add-cart by KID)

Then runs `excel__recalculate_formulas` on final workbook.

Meaning:
- This stage is the **core bronze→silver aggregation** stage already SQL-heavy.
- PT sheets are intermediate fact tables for report formulas.

## Stage S4: Group-by-SPU outputs
File: `step4-groupbyspu.json`

- Reads `周数据-all`
- SQL group-by for selected shops/categories
- Writes grouped outputs to category/group sheets

Meaning:
- This stage is report slicing/distribution by team/category/shop.

## 3) Sample output workbook structure (ground truth)

Workbook: `COPY-v5-0126--verify.xlsx`

Key sheets mapped from workbook XML:
- raw/source-like: `大盘`, `广告`, `联盟`, `ERP`, `订单`
- PT intermediate: `大盘-PT`, `广告-PT`, `联盟-PT`, `ERP-主货号-PT`, `订单中间表-PT`, `商品编号-PT`
- report: `周数据-all`, `周数据-小组`, `链接数据统计`, team split sheets (`马来家具鸿耀` ...)

Observed characteristics:
- PT sheets are mostly value tables (no formulas in sampled PT tabs)
- final report sheets are formula-heavy:
  - `周数据-all`: 27,120 formulas
  - `链接数据统计`: 49,572 formulas
  - `周数据-小组`: 10,061 formulas
- dominant formula pattern is lookup/aggregation against PT tabs:
  - `IFERROR(XLOOKUP(...),0)`
  - `SUMIFS(...)`
  - `VLOOKUP(...)`

Representative formulas found:
- `IFERROR(_xlfn.XLOOKUP($D2,'大盘-PT'!$F:$F,'大盘-PT'!D:D),0)`
- `IFERROR(_xlfn.XLOOKUP($D2,'订单中间表-PT'!$F:$F,'订单中间表-PT'!D:D),0)`
- `SUMIFS(大盘!J:J,大盘!B:B,D2,大盘!E:E,"-")`
- `_xlfn.XLOOKUP(D2,'商品编号-PT'!A:A,'商品编号-PT'!D:D)`

Conclusion:
- Current final outputs are generated by:
  1) SQL aggregations into PT tabs
  2) key expansion/autofill
  3) large formula networks that consume PT/raw tabs

## 4) Drawbacks in the current workflow data pipeline

1. **Workflow orchestration is too heavy for stable ETL**
   - multiple nested workflow calls, hard to reason/debug end-to-end

2. **Data lineage is implicit**
   - transformations live in JSON tool-call payloads and copied SQL strings
   - difficult to version/control model-by-model changes

3. **Formula explosion at report layer**
   - tens of thousands of formulas in final sheets increase recalc/memory cost

4. **Mixed responsibilities in workbook**
   - raw, PT, and report sheets coexist in one artifact, coupling ETL and presentation

5. **Verification is not standardized**
   - currently no strict phase gates with fixed parity KPIs

## 5) SQL-equivalent mapping for formula-heavy logic (migration principle)

Keep formula compatibility where business users need Excel behavior, but push high-volume logic to SQL.

## Formula families and SQL equivalents

1. `XLOOKUP(key, PT!K:K, PT!V:V)`
   - SQL: `LEFT JOIN pt_table USING(key)` and select joined columns

2. `SUMIFS(range, key_col, key, cond_col, cond)`
   - SQL: `SUM(CASE WHEN key_col=... AND cond_col=... THEN range ELSE 0 END)`

3. `IFERROR(expr, 0)`
   - SQL: `COALESCE(expr, 0)`

4. `VLOOKUP(key, table, col, 0)`
   - SQL: exact-match join + projected column

Migration rule:
- PT/gold pre-aggregation in DuckDB
- keep only presentation-level formulas in export templates

## 6) Verification blueprint for migration parity

All migration phases must satisfy both data parity and formula parity.

## A. Data parity checks (mandatory)

For key target sheets (`周数据-all`, `周数据-小组`, `链接数据统计`):

1. Row-count parity
2. Key uniqueness parity (`KID`, item id)
3. Numeric metric parity (sum/min/max across critical KPI columns)
4. Null/blank distribution parity

Suggested tolerance:
- counts/keys: exact
- floats: absolute diff <= 1e-6 or business-defined tolerance

## B. Formula parity checks (mandatory)

For formula-heavy report sheets:

1. Formula preservation check
   - compare formula text in sampled cells
2. Recalc result check
   - trigger recalc and compare resulting values
3. Cross-sheet dependency check
   - mutate one PT source value and verify expected downstream formula cells change

## C. Workbook structure parity

1. Required sheet names exist
2. Required named ranges/filter headers preserved
3. Key template ranges remain stable for autofill/export

## 7) Minimal verification dataset and regression cases

Use a stable fixture dataset with:
- MY and TH shops
- multiple SPU categories
- edge cases: missing values, canceled orders, invalid SKU refs

Regression scenarios:
1. full weekly run
2. single-shop partial run
3. no-data category run
4. source correction rerun

Expected outputs:
- PT tabs updated
- report tabs regenerated
- formula recalc stable

## 8) Success criteria definition (high level)

A migration step is considered successful only if:
1. output values match baseline parity checks
2. formula behavior remains correct for retained formulas
3. runtime cost improves (time/memory)
4. rollback path works via feature flag
