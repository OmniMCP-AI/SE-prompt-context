# E-commerce Daily Anomaly Checking

Use this skill to triage MaybeAI ecommerce daily workbook outputs without confusing pipeline artifacts, workbook template defects, and true business anomalies.

This skill is intended to stand on its own for workbook review. If the internal handoff docs are available, they can be used as deeper references, but do not require the user to read them before applying the checks.

## Scope

Default scope is `org_id=None` for MaybeAI's 七木 ecommerce flow, especially outputs from:

- `POST /api/v1/excel/process_daily_data`
- Daily workbooks named like `日数据-*` / `链接数据*`
- 七木 workbooks that retain weekly-era sheet names like `周数据-*`

The default daily run uses 11 Shopee shops and treats `广告`, `订单`, and `联盟` as Shopee MCP categories. It also consumes selected ZIP contents. Do not assume every sheet in an uploaded ZIP is supposed to appear in the final workbook.

## Workbook Layers

- Raw/source-like: `大盘`, `广告`, `联盟`, `ERP`, `订单`, `日期表`
- Daily report: `日数据-all`, `链接数据统计`
- Weekly/legacy report aliases: `周数据-all`, `周数据-小组`
- PT/intermediate, when present: `大盘-PT`, `广告-PT`, `联盟-PT`, `ERP-主货号-PT`, `订单中间表-PT`, `商品编号-PT`
- Presentation/split: `日数据-小组`, `日数据-家具家居`, `马来家居`, `马来家具鸿耀`, `马来家具雪`, `泰国家具林琳`, `泰国家居`, `链接数据-*`

Report sheets are formula-heavy. PT sheets are normally value tables produced by SQL-like aggregation. Final report values are commonly generated by key expansion plus formula families such as `XLOOKUP`, `SUMIFS`, `VLOOKUP`, and `IFERROR(...,0)`.

When reviewing business results, use cached cell values. Use formula text only to trace lineage, detect template defects, or confirm that report formulas still point to the expected PT/raw sheets.

## Daily source ownership

- Shopee MCP primary sources: `广告`, `订单`, `联盟`
- ZIP primary sources: `商业分析`, `ERP`, `入库单明细`
- ZIP `订单`, `联盟`, `广告`, and `库存报表` are not daily primary sources under the default flow. Their absence from daily output is usually expected.
- `联盟` MCP failure can be ignored by the daily pipeline. `广告` or `订单` MCP failure should abort the import.
- Workbook output may still contain non-Shopee stores in downstream report sheets from formulas, prior target workbook state, or non-Shopee ZIP-derived facts. Judge mismatches against source ownership before calling them data loss.

## Daily filtering and normalization rules

- `data_date` defaults to Beijing-time yesterday (`T-1`) if not supplied.
- `日期表` is the first place to confirm the actual report date used by formulas.
- `订单` is filtered by each row's order creation time. Recognized source date fields include Chinese headers such as `创建时间` / `下单时间` and English variants such as `order create`, `creation date`, or `creation time`.
- `广告` and `商业分析` are filtered by date parsed from the file name, not by row-level dates. If no date can be parsed from the file name, the file may bypass that date filter.
- Daily `商业分析` is narrowed to the single `data_date`.
- `广告` CSV files may skip the first 7 rows before reading headers.
- Legacy no-header `广告` files may have synthetic headers injected.
- `商业分析`, `联盟`, and `广告` may have English headers translated to target Chinese headers, currency suffixes stripped, and unrecognized columns dropped.
- Normal category merges prepend `站点`, `店铺`, and `店铺ID`, then align rows to template or target workbook headers.
- `ERP` and `入库单明细` merge the first worksheet of each file, keep only the first header row, and skip fully blank rows.
- Daily output only uploads sheets visible in the template or target header set. A sheet can exist in intermediate processing but not be written if it is outside that set.

## ERP cost backfill semantics

For default 七木 daily runs, `ERP.采购成本` may be backfilled from `入库单明细` after ZIP merge. This is expected behavior, not automatic evidence of tampering.

- Only blank or zero `ERP.采购成本` rows are backfilled.
- Primary match key: `ERP.仓库 + ERP.出库SKU` to `入库单明细.收货仓 + 入库单明细.商品编码`.
- Thai warehouse rows can fall back to same-SKU Thai-warehouse candidates when exact warehouse matching fails.
- `状态 = 已关闭` inbound rows are excluded.
- Candidate priority: prefer `在途`; otherwise use `已完成`.
- Time choice: earliest `创建时间` for `在途`, latest `创建时间` for `已完成`.
- Cost source: `入库单明细.采购费用`, parsed as CNY.
- Currency conversion defaults: `MYR = 1 / 1.65`, `THB = 1 / 0.21`, `CNY/RMB = 1.0`.
- Written value is negative: `- round(采购费用CNY * 汇率 * 出库SKU销量, 2)`.

Negative `采购成本` caused by this backfill is expected. Investigate only if the match source is missing, the sign convention differs inside the same workbook, currency is unknown at scale, or `出库SKU销量` cannot be parsed for affected rows.

## Triage order

1. Confirm report date and run mode from `日期表`, file name, user request, or API parameters.
2. Identify the workbook layer for each suspicious cell: raw/source, PT/intermediate, report, or split/presentation.
3. Read cached values for business interpretation. Load formula text separately only when tracing lineage or checking template defects.
4. Verify source coverage by ownership:
   - Shopee MCP: `广告`, `订单`, `联盟`
   - ZIP daily: `商业分析`, `ERP`, `入库单明细`
5. Check date filtering before flagging missing rows:
   - `订单`: row-level order creation time
   - `广告` / `商业分析`: file-name date
   - `商业分析`: single-day daily window
6. Check joins and key generation:
   - SKU/report keys often combine store, `SPU分类`, and `主货号` concepts.
   - Link reports are driven by unique `商品编号`.
   - Missing `主货号`, `SPU分类`, `商品编号`, or inconsistent store IDs are higher-signal than downstream zero formulas.
7. Inspect PT sheets or lookup sources before blaming final report formulas.
8. Inspect presentation and split sheets last. Many visible errors there are template hygiene issues rather than source data defects.

## What is usually a real issue

- `广告` rows missing `主货号` or `SPU分类` at scale.
- `订单` rows outside the requested day when their creation time should have been filtered out.
- Requested-day `订单` rows missing from MCP-derived output for default Shopee shops.
- `广告` or `商业分析` files with valid date-stamped names that should be in the daily window but are absent.
- `ERP` rows that fail join/backfill against `入库单明细`.
- Live SKU rows with `SPU分类 = 0` or blank when they are not `TEST` / `测品` / auxiliary barcode rows.
- `毛利润率 < 5%` or materially high `营销占比` on active SKUs.
- Report keys that are duplicated in PT/report sheets when the sheet expects one row per key.
- Formula lookup keys that do not exist in their PT/source sheet even though the raw source has usable data.

## What is often expected

- Daily ZIP `订单`, `联盟`, `广告`, or `库存报表` files not appearing as daily primary output.
- `联盟` absent or stale when the daily MCP affiliate fetch failed but the pipeline continued.
- `sales > 0` with `order = 0` for non-Shopee sources (`TikTok`, `Lazada`) or auxiliary barcode SKUs.
- Negative `自然访客` in `链接数据统计` when it is derived from `访客 - 广告点击`.
- Negative `采购成本` in `ERP` when backfilled from `入库单明细`.
- Blank placeholder sheets with only headers when a template sheet had no daily data.
- Columns from source files missing in output because target/template headers did not include them.
- Files scanned in ZIP logs but skipped because they are temp files, archive-only directories, ad-group files, unsupported formats, or names that do not match category rules.

## What is usually a template defect

- `#DIV/0!` in `日数据-小组` or split sheets.
- Divide-by-zero formulas like `=E2/G2` on zero rows.
- Repeated or extra split-sheet columns that do not change the main raw or PT sheets.
- Formula ranges that point to obsolete weekly sheet names when daily aliases are expected, unless the workbook intentionally uses weekly-era names.
- Formula text missing in copied-down report rows while surrounding rows have equivalent formulas.

## Verification checks

For a workbook-level anomaly review, include the highest-signal checks that apply:

- Required sheet presence: raw/source-like sheets, report sheets, `日期表`, and any PT sheets used by formulas.
- Row counts by layer: raw/MCP source, PT sheets, main report, split reports.
- Key uniqueness: SKU key in `日数据-all` or `周数据-all`; `商品编号` in `链接数据统计`.
- Numeric sanity: sum/min/max of orders, sales, ad spend, affiliate cost, ERP sales/cost, gross margin, marketing ratio.
- Null/blank distribution: `主货号`, `SPU分类`, store ID, SKU, item ID, currency, cost.
- Formula preservation: sampled formula text in copied ranges, especially lookup and ratio columns.
- Recalc consistency: cached value agrees with formula intent after source values are known.
- Cross-sheet lineage: suspicious final cells trace to the expected PT/raw sheet and key.

## Output format

Return findings as:

- `type`: `expected` | `template-defect` | `pipeline-mismatch` | `business-risk`
- `sheet`
- `row`
- `key`
- `why`
- `evidence`: source sheet/value/formula/date-filter fact, when useful

Prefer the main report sheets for business review:
- `日数据-all` for SKU-level risk
- `链接数据统计` for product-linking/traffic risk
- `日数据-小组` only for presentation errors
- `周数据-all` / `周数据-小组` when the workbook uses weekly-era sheet names

Keep findings concise and separate expected artifacts from defects. If the evidence depends on a pipeline assumption, state the assumption explicitly, for example: "default `org_id=None` daily flow; ZIP ads are not daily primary input."
